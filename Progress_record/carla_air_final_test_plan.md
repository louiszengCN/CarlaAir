# CARLA-Air 开源前测试计划（最终融合版）

**版本**：v3.0（融合版）  
**基于**：技术报告测试建议方案 v2.0 + 团队测试清单 v1.0  
**执行对象**：AI Code Agent  
**日期**：2026-03-11

---

## 关于本文档

本文档由两份独立测试计划融合而成：

- **方案 A**（技术报告测试建议 v2.0）：从审稿人、自动驾驶研究者、ROS 开发者等六类视角出发，侧重**深度**——官方示例脚本逐一兼容性矩阵、传感器数值精度验收标准、Traffic Manager 阈值边界量化、ROS 话题消息类型规范、性能基准补全等。
- **方案 B**（团队测试清单 v1.0）：从工程可用性出发，侧重**广度**——安装流程验证、重复启动稳定性、异常恢复、非法输入处理、RViz 可视化、TF 坐标树、文档可读性等。

两份方案高度互补，**无实质性冲突**，融合后覆盖面显著优于任意单份。融合策略如下：

| 维度 | 方案 A 优势 | 方案 B 优势 | 融合处理 |
|------|------------|------------|----------|
| 官方示例脚本 | 逐脚本兼容性矩阵（CARLA 12 个 + AirSim 35 个） | 仅提及"跑通常用 example" | 采用方案 A 的完整脚本清单 |
| ROS 测试 | 话题名称/消息类型/验收标准完整规范 | 新增 RViz 可视化、TF 坐标树、rosbag | 全量合并，RViz/TF 为方案 B 独有新增 |
| 安装与启动 | 仅提及"新用户独立完成" | 干净环境安装、重复启动、headless 启动 | 采用方案 B 的结构化安装测试 |
| 异常恢复 | 边界条件与异常处理（较简略） | Client 掉线恢复、仿真器重启、非法输入、顺序错误操作 | 采用方案 B 的完整恢复测试章节 |
| 稳定性测试 | 6 小时连续 + 压力测试 | 中时长 + 长时 + 高负载 actor/sensor + 多 ROS 节点并发 | 全量合并，补充"多 ROS 节点并发"场景 |
| 数值精度验收 | 详细（位置误差 < 0.5 m、深度误差 < 5% 等） | 定性描述为主 | 保留方案 A 的量化验收标准 |
| 文档测试 | 文档流程验证、已知限制准确性 | 新增"常见报错可理解性"、"ROS 支持范围说明" | 全量合并 |

> **执行建议**：本文档的所有测试项均应由 AI agent 执行。鉴于两份方案高度互补，**建议全量执行融合后的测试计划**，而非选择其中一份。P0 级测试为发布阻断项，必须全部通过；P1 级强烈建议在发布前完成；P2 级可在 v1.0 发布后迭代。

## 第一章：安装与启动测试

> **来源**：方案 B 第 1 章（主体）+ 方案 A 文档验证章节（补充量化标准）

### T1.1 干净环境安装

**测试内容**：在全新 Ubuntu 20.04 环境中，严格按照 README 文档从零开始完成安装与配置，不依赖任何隐式前置条件。

**执行步骤**：
1. 从 GitHub 克隆仓库
2. 按 README 逐步执行所有安装命令
3. 记录每一步的执行结果和耗时
4. 记录任何文档未提及但实际需要的步骤

**验收标准**：30 分钟内完成安装并成功运行第一个示例脚本；所有 README 命令可直接执行，无需额外隐式操作。

**优先级**：P0

---

### T1.2 基础启动验证

**测试内容**：启动 CARLA-Air 并验证 CARLA（端口 2000）和 AirSim（端口 41451）均进入可连接状态。

**执行步骤**：
1. 执行启动命令
2. 等待启动完成
3. 尝试通过 Python 连接 CARLA client（`carla.Client('localhost', 2000)`）
4. 尝试通过 Python 连接 AirSim client（`airsim.MultirotorClient()`）
5. 验证两个连接均成功

**验收标准**：两个 client 均连接成功，无初始化错误日志；启动时间 < 5 分钟。

**优先级**：P0

---

### T1.3 重复启动稳定性

**测试内容**：连续执行 5 次"启动 → 连接验证 → 关闭"循环，检查是否存在端口残留、资源未释放等问题。

**执行步骤**：
1. 重复执行启动-验证-关闭循环 5 次
2. 每次启动前检查端口 2000 和 41451 是否已释放
3. 记录每次启动时间和是否出现异常

**优先级**：P0

## 第二章：CARLA 基础能力测试

> **来源**：方案 B 第 2 章（结构）+ 方案 A 第二章（深度补充）

### T2.1 CARLA 连接与基础查询

**测试内容**：Python client 连接 CARLA，读取 world、map、blueprints 等基础信息。

**执行步骤**：
1. 建立 CARLA client 连接，设置超时为 10 秒
2. 调用 `client.get_world()` 获取 world 对象
3. 调用 `world.get_map()` 获取地图信息，读取地图名称
4. 调用 `world.get_blueprint_library()` 获取蓝图库，统计蓝图数量
5. 调用 `client.get_server_version()` 验证版本信息

**验收标准**：连接成功，蓝图数量 ≥ 220，版本号为 0.9.16，无 RPC 超时。

**优先级**：P0

---

### T2.2 地图与世界控制

**测试内容**：加载地图、切换地图、tick 推进、同步模式设置。

**执行步骤**：
1. 验证当前地图名称
2. 设置同步模式（`fixed_delta_seconds=0.05`），调用 `world.tick()` 10 次，验证仿真时间精确推进
3. 切换至另一张地图（如从 Town01 切换至 Town03），验证切换成功
4. 切换后重新连接，验证功能完整
5. 调用 `world.get_map().get_topology()` 获取路网数据，验证不为空

**验收标准**：同步模式下每次 tick 仿真时间精确递增 0.05 秒；地图切换后功能完整；路网数据非空。

**优先级**：P0

### T2.3 全量地图启动验证

**测试内容**：依次以全部 13 张地图启动 CARLA-Air，验证每张地图均可正常使用。

**执行步骤**：对以下 13 张地图逐一执行：Town01、Town02、Town03、Town04、Town05、Town06、Town07、Town10HD、Town11、Town12、Town13、Town15、CarlAirport

### T2.4 车辆生命周期管理

**测试内容**：全量 41 种车辆蓝图的 spawn/destroy，以及生命周期边界条件。

**执行步骤**：
1. 遍历蓝图库中所有 `vehicle.*` 类型蓝图（预期 41 种），逐一 spawn 并 destroy，记录每个蓝图的结果
2. 一次性 spawn 50 辆车，记录 VRAM 增量，然后全部 destroy
3. 循环 spawn/destroy 同一蓝图 1000 次，监控 VRAM 是否泄漏
4. 销毁 Actor 后尝试操作旧 ID，验证返回合理错误

**验收标准**：41 种蓝图全部 spawn 成功；1000 次循环后 VRAM 增长 < 50 MiB；旧 ID 操作返回明确错误而非崩溃。

**优先级**：P0（全量蓝图）/ P1（循环泄漏）/ P2（ID 重用）

**验收标准**：52 种蓝图全部 spawn 成功；行人 AI 导航成功率 > 80%。

### T2.9 传感器系统全面测试

**测试内容**：地面车辆全类型传感器的数据质量与同步精度。

**执行步骤**：

*数据质量验证（逐类传感器）：*
1. **RGB 相机**：在晴天/雨天/夜间三种天气下采集图像，人工抽检有无伪影
2. **深度相机**：在已知距离（5 m、10 m、20 m、50 m）场景下采集深度图，计算误差
3. **语义分割**：采集语义图，抽检车辆/道路/建筑/天空/行人标签正确性
4. **实例分割**：多辆车同框，验证每辆车实例 ID 唯一
5. **LiDAR（普通）**：采集点云，验证格式（XYZI）、点数与配置一致
6. **语义 LiDAR**：采集语义点云，验证每个点的语义标签
7. **Radar**：在有运动目标的场景下验证速度/距离字段
8. **IMU**：静止车辆采集 1 分钟，分析偏置
9. **GNSS**：对比 GNSS 输出与 `actor.get_transform()` 的位置
10. **碰撞传感器**：以 10/30/60 km/h 撞击静态障碍物，验证事件触发
11. **车道入侵传感器**：跨越实线/虚线车道线，验证事件触发

*多传感器同步精度验证：*
12. 在同步模式下同时挂载 RGB + 深度 + 语义 + LiDAR + IMU + GNSS，验证同一 tick 内所有传感器帧号一致
13. 以 30 FPS 连续采集 10 分钟，统计丢帧率

*传感器生命周期：*
14. 反复创建、销毁、重建传感器 100 次，验证无残留 callback 或脏状态

**验收标准**：深度误差 < 5%；语义标签与视觉内容一致；同一 tick 内所有传感器帧号相同；丢帧率 < 0.1%；传感器重建后数据正常。

**优先级**：P0（RGB/深度/语义/LiDAR/同步精度）/ P1（其余传感器、丢帧率）/ P2（实例分割）
